# v1.27.0 Migration Guide — Tenant-agnostic `@sanamyvn/ai-ts`

> Audience: consumers upgrading from v1.26.x to v1.27.0.
>
> **Note on versioning:** v1.27.0 contains breaking changes despite the minor
> version bump. The release automation classified the change as `feat` rather
> than a major bump; the breaking content below is real. Pin and migrate
> deliberately.
>
> This guide covers every breaking change and gives you copy-paste recipes
> to update your call sites.

## Breaking changes at a glance

| Area           | Change                                                                                                                                                   |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| DB schema      | `ai_sessions.tenant_id` column removed                                                                                                                   |
| DB schema      | `ai_session_messages.tenant_id` column removed                                                                                                           |
| Parameters     | `tenantId` removed from `CreateConversationCommand`, `ListSessionsQuery`, `CountMessagesFilter`, `StartSessionInput`, and all related Zod schemas / DTOs |
| Output shape   | `Session.tenantId` field removed — `GET /ai/sessions/:id` and list responses no longer include `tenantId`                                                |
| Query rename   | `CountMessagesByTenantQuery` renamed to `CountMessagesQuery`                                                                                             |
| Dispatch type  | `'ai.session.countMessagesByTenant'` removed; use `'ai.session.countMessages'`                                                                           |
| Method rename  | `SessionAppService.countMessagesByTenant` → `countMessages`                                                                                              |
| Middleware key | `MiddlewareConfig.countMessagesByTenant` → `MiddlewareConfig.countMessages`                                                                              |

## 1. Database migration

Run the following DDL against every schema (including per-tenant schemas)
that contains `ai_sessions` and `ai_session_messages` before deploying v1.27.0:

```sql
ALTER TABLE ai_sessions DROP COLUMN tenant_id;
ALTER TABLE ai_session_messages DROP COLUMN tenant_id;
```

Adjust for your host app's actual schema-per-tenant DDL convention (e.g.
`ALTER TABLE tenant_a.ai_sessions DROP COLUMN tenant_id`).

ai-ts does not ship migration files. DDL is your responsibility.

## 2. Find / replace recipe

Apply these changes across your codebase in order:

### 2a. Remove `tenantId` from call sites

Delete any line that passes `tenantId` into an ai-ts command, query, or
DTO. Examples of lines to delete:

```
tenantId: this.asyncCtx.require(TENANT_ID),
tenantId,
tenantId: input.tenantId,
```

For non-strict Zod schemas (`ListSessionsQuery`, `CreateConversationCommand`,
`sessionListQueryDto`, `createConversationDto`), the field is silently
stripped if you leave it — but the TypeScript compiler will error on it.
Remove it.

For `countMessagesQueryDto` (`GET /ai/sessions/message-events/count`),
`tenantId` was previously **required**. Remove it from both the client
request and any server-side DTO construction.

### 2b. Rename the count-messages query

| Find                                     | Replace                          |
| ---------------------------------------- | -------------------------------- |
| `CountMessagesByTenantQuery`             | `CountMessagesQuery`             |
| `'ai.session.countMessagesByTenant'`     | `'ai.session.countMessages'`     |
| `.countMessagesByTenant(`                | `.countMessages(`                |
| `MiddlewareConfig.countMessagesByTenant` | `MiddlewareConfig.countMessages` |

### 2c. Update import paths

```typescript
// Before
import { CountMessagesByTenantQuery } from '@sanamyvn/ai-ts/business/session/client/queries';

// After
import { CountMessagesQuery } from '@sanamyvn/ai-ts/business/session/client/queries';
```

### 2d. Remove `tenantId` from `CountMessagesQuery` payload

```typescript
// Before
new CountMessagesByTenantQuery({
  tenantId,
  purposePrefix: 'ta-chat:',
  sentAtGte: periodStart,
  sentAtLt: periodEnd,
});

// After
new CountMessagesQuery({
  purposePrefix: 'ta-chat:',
  sentAtGte: periodStart,
  sentAtLt: periodEnd,
});
```

### 2e. Remove `session.tenantId` reads

Any code that reads `session.tenantId` will produce a TypeScript compile
error after upgrading. Delete those reads. Tenant identity is now the host
app's responsibility — it is in scope on the host side (e.g. from
`asyncCtx`) and does not need to round-trip through the session object.

## 3. Tenant isolation going forward

v1.27.0 shifts isolation responsibility to the host app. See the
[Multi-tenancy section in the README](../../README.md#multi-tenancy) for
the `search_path` contract and the three-step setup pattern.

In brief: set `SET LOCAL search_path = <tenant_schema>, public` inside
your request-scoped transaction before calling any ai-ts method. ai-ts
runs all queries through the injected `AI_DB` client and inherits
whatever `search_path` is active.

## 4. Verification checklist

Before deploying:

- [ ] `pnpm check-types` passes with no `tenantId`-related errors
- [ ] `grep -r "tenantId" src/ --include="*.ts"` in your host app returns
      only your own code (not ai-ts imports)
- [ ] DB migration has been applied to every schema (including staging)
- [ ] Integration tests that previously passed `tenantId` have been updated
