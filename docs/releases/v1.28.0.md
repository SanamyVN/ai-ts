# `@sanamyvn/ai-ts` v1.28.0 — Paginated Responses Now Include `total`

Both paginated responses — `ListSessionsQuery` and `GetSessionMessagesQuery` —
now carry an inline `total: number` field. Downstream HTTP layers (digi-talk
first) can build the canonical `{ data, meta: { total, page, perPage, totalPages } }`
envelope from the SDK without a separate count round trip and without breaching
the layer boundary.

See [docs/design/paginated-total-counts.md](../design/paginated-total-counts.md)
for the full design rationale.

## 1. Why v1.28.0 is a minor, not a major

Despite containing a field rename and a page-indexing behavior change, this
release ships as a minor (per [design doc §7.1](../design/paginated-total-counts.md#71-version)):

- The rename touches one field on one response shape.
- The only external consumer team (digi-talk) is the feature requester and is
  adopting both sides simultaneously; there is no other consumer to protect.
- All in-package call sites are updated in the same commits (see §4.5 of the
  design doc for the complete audit).
- A semver-major bump would force every v1.x consumer to re-pin without gaining
  any additional protection — the package has one downstream team and no
  maintenance branch.

If a future audit surfaces an unknown external consumer of `.messages`, that is
a real cost and should be revisited before tagging. Flagged explicitly here so
the question does not get lost.

## 2. Acknowledged break — rename `messages → items`

`GetSessionMessagesQuery` response field `messages` is renamed to `items`.

**Before (v1.27.0):**

```ts
const { messages, page, perPage } = await mediator.send(
  new GetSessionMessagesQuery({ sessionId, page: 1, perPage: 50 }),
);
```

**After (v1.28.0):**

```ts
const { items, page, perPage, total } = await mediator.send(
  new GetSessionMessagesQuery({ sessionId, page: 1, perPage: 50 }),
);
```

**One-line migration:** rename `.messages` to `.items` in your destructures and
property accesses.

For a grep recipe that catches both destructuring and property-access patterns,
see the [adoption guide §3](./v1.28.0-adoption-guide.md#3-rename-migration-messages--items).

**Why rename:** aligns both paginated responses on the same shape
`{ items, page, perPage, total }` so generic pagination helpers can work against
either without branching. The requesting team's canonical envelope also uses
`items`. Keeping `messages` alongside `items` was rejected — it doubles the
payload and defers the rename without reducing the migration cost.

## 3. Acknowledged break — page-indexing fix

`GetSessionMessagesQuery` `page: 1` now returns the first slice of the thread.

In v1.27.0, `GetSessionMessagesQuery` enforced `page >= 1` at the public
boundary, but `MastraMemoryAdapter` forwarded the page number unchanged to
Mastra's `recall()`, which is **0-indexed**. As a result, a public call with
`page: 1` read Mastra's second page. Mastra page 0 — the conversation's
earliest messages under `createdAt ASC` order — was unreachable from the SDK.

The fix subtracts 1 in `MastraMemoryAdapter.getMessages` before forwarding to
`recall()`. The response continues to echo the **public** (1-based) page number
the caller passed in. See [design doc §2](../design/paginated-total-counts.md#section-2--getsessionmessagesquerytotal)
for the full technical explanation.

**Impact on callers:** any v1.27.0 caller that paginated messages from `page: 1`
was silently missing the earliest messages in the thread. After this fix they
will start receiving those messages. The change is in the bug-fix direction —
callers gain access to data they could not previously reach.

If you worked around the bug by starting pagination from `page: 2`, remove that
workaround. See [adoption guide §4](./v1.28.0-adoption-guide.md#4-page-indexing-fix).

## 4. Additive change — `total` on `ListSessionsQuery`

`ListSessionsQuery` response gains `total: number`. Every existing destructuring
of `{ items, page, perPage }` keeps compiling and behaving identically — `total`
is an additional field.

`total` is the count of `ai_sessions` rows that match the same filter the caller
passed, without applying `page`/`perPage`. It satisfies five invariants documented
in [design doc §3](../design/paginated-total-counts.md#section-3--filter-parity--pagination-invariants):
non-negative, `>= items.length` for any single page, filter-parity (same
`WHERE` clause as the page rows), page-walk completeness, and correct value on
past-the-end pages.

Example response payload:

```json
{
  "items": [
    { "id": "sess_abc", "messageCount": 4, "purpose": "ta-chat:ta-uuid-1" }
  ],
  "page": 1,
  "perPage": 50,
  "total": 137
}
```

`totalPages` is not included in the SDK response — derive it in the HTTP layer:
`Math.ceil(total / perPage)`. When `total === 0`, produce `totalPages: 0`.
See [adoption guide §2](./v1.28.0-adoption-guide.md#2-worked-example--building-data-meta) for
a complete worked example.

**SQL plan.** The count is computed with `COUNT(*) OVER ()` in the same query
that fetches the page — one round trip in the common case. When the page returns
zero rows (past-the-end request), a follow-up `SELECT COUNT(*)` is issued with
the same filter. See [design doc §1](../design/paginated-total-counts.md#section-1--listsessionsquerytotal)
for details.

**Performance baseline.** The `COUNT(*) OVER ()` approach was chosen because it
avoids a separate count round trip and lets Postgres reuse the existing index
scan on the dominant filter predicate (typically `started_at`; design doc §5.2).
The §5.2 performance assumption is verified by
`src/__tests__/integration/session/session-list-pagination.spec.ts`, which
asserts a 500 ms wall-clock ceiling and an index scan on the dominant filter
predicate over a 10k-row fixture. The test runs in CI on every commit;
regression fails the build.

## 5. Additive change — `total` on `GetSessionMessagesQuery`

`GetSessionMessagesQuery` response gains `total: number`. This is the count of
messages Mastra's `recall()` would return for the given `sessionId` across all
pages.

Example response payload:

```json
{
  "items": [
    { "id": "msg_1", "role": "user", "content": "Hello" }
  ],
  "page": 1,
  "perPage": 50,
  "total": 312
}
```

**Important distinction:** this `total` is **not** the same number as
`SessionSummary.messageCount`.

| Field | Source | Counts |
|-------|--------|--------|
| `GetSessionMessagesQuery` response `total` | Mastra thread transcript (`recall()`) | Every message role the thread contains — user, assistant, system, tool |
| `SessionSummary.messageCount` | Local `ai_session_messages` ledger | Successful user-submitted messages only — used for billing |

The two numbers diverge by design. Do not substitute one for the other.

`total` is plumbed directly from Mastra's `PaginationInfo.total` field that
`recall()` already returns on every call — no new Mastra round trip, no
`COUNT(*)` of our own. See [design doc §2](../design/paginated-total-counts.md#section-2--getsessionmessagesquerytotal).

## 6. Non-changes

The following are explicitly **unchanged** in v1.28.0:

- `SessionSummary.messageCount` — still the user-submitted message count from
  the local `ai_session_messages` ledger. Semantics, naming, and billing
  behavior are identical to v1.27.0.
- `CountMessagesQuery` — unchanged. Still the recommended billing
  aggregate for period-bounded message counts.
- The `ai_session_messages` ledger table — unchanged. No new columns, no schema
  migration required for v1.28.0.
- All other query types (`GetSessionQuery`, `CreateSessionQuery`,
  `UpdateSessionTitleQuery`, `DeleteSessionQuery`,
  `RecordMessageEventQuery`) — unchanged.

**DB migration: one additive index.** v1.28.0 adds a btree index on
`ai_sessions.started_at` (`ai_sessions_started_at_idx`) so the `COUNT(*) OVER ()`
list path can avoid a sequential scan once a deployment's session table grows
beyond a few thousand rows. The migration is additive and safe to run online;
existing rows and columns are untouched. No application-side change is required
— the index is purely a planner aid. See [design doc §6.2](../design/paginated-total-counts.md#62-test-plan)
for the scale-baseline test that motivates it.

## 7. Why no standalone count queries

A standalone `CountSessionsQuery` is **deferred**, not rejected on principle.

Inline `total` covers the current need (building `{ data, meta }` in one SDK
call without a second round trip). A future caller that legitimately needs a
count without fetching a page at all — e.g. an admin widget showing "this user
has N sessions" — is a fine motivator to add `CountSessionsQuery` then. The
inline-`total` path does not preclude it; the two coexist cleanly.

See [design doc §5.1](../design/paginated-total-counts.md#51-inline-total-on-the-list-response-not-a-separate-countquery)
for the full decision record.

## 8. Mixed-version deployments are not supported

There is no v1.27.0 maintenance branch. SDK and server must move to v1.28.0
together.

The `RemoteSessionMediator` deserializes responses using `messageListClientSchema.parse(...)`.
That Zod schema now requires `items` (not `messages`) and `total`. A v1.28.0
SDK pointed at a v1.27.0 server will receive a `{ messages, page, perPage }`
response body and the Zod parse will throw — the mediator will not silently
return stale-shaped data.

The reverse is also true: a v1.27.0 SDK pointed at a v1.28.0 server will
receive `{ items, page, perPage, total }`, which fails the v1.27.0 schema that
expects `messages`. Flip both sides in the same deploy window.
