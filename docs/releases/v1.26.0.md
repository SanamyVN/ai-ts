# `@sanamyvn/ai-ts` v1.26.0 — Subscription & Entitlement Support

This release adds the metric and query surface the digi-talk Subscription &
Entitlement Engine needs to bill on real chatbot-message volume per period
without proxying.

## 1. Required DB migration

Run before deploying the upgraded application:

```sql
CREATE TABLE ai_session_messages (
  id          text        PRIMARY KEY,
  session_id  text        NOT NULL,
  tenant_id   text        NOT NULL,
  purpose     text        NOT NULL,
  sent_at     timestamptz NOT NULL DEFAULT NOW()
);
```

## 2. Recommended indexes (not shipped)

```sql
CREATE INDEX ai_session_messages_session_idx
  ON ai_session_messages (session_id);

CREATE INDEX ai_session_messages_tenant_sent_idx
  ON ai_session_messages (tenant_id, sent_at);

CREATE INDEX ai_session_messages_tenant_purpose_sent_idx
  ON ai_session_messages (tenant_id, purpose, sent_at);
```

- `session_idx` is **required** for acceptable session-list performance —
  `SessionRepository.list` makes a per-page `countBySession` query that
  scans this index.
- The two `(tenant_id, ...)` composites are billing-scan helpers; recommended
  once a tenant's scan set exceeds a few thousand rows.

## 3. Breaking changes

- **`ListSessionsQuery` response shape**: `SessionSummary[]` →
  `{ items: SessionSummary[]; page: number; perPage: number }`.
- **`ListSessionsQuery` payload**: `page` and `perPage` are now required
  (`page` ≥ 1, `perPage` 1–500).
- **`GetSessionMessagesQuery` payload**: `page` must be ≥ 1, `perPage`
  must be 1–500 (previously unconstrained).

## 4. New HTTP routes (additive)

Consumers using the local mediator are unaffected. Remote-mediator users
pick these up automatically.

- `POST /ai/sessions/:id/message-events` → `204 No Content`
- `GET  /ai/sessions/message-events/count` → `200 { count: number }`

Query string for the aggregate: `tenantId` (required), `purpose`,
`purposePrefix`, `sentAtGte` (ISO 8601), `sentAtLt` (ISO 8601). Same zod
refinements as the mediator payload.

## 5. Backfill semantics

`ai_session_messages` ships empty. Pre-existing sessions therefore report
`messageCount: 0` until their next user message; aggregate counts over
historical periods are zero. The first full period after deploy is the
first accurate one. Communicate this to tenants sensitive to billing-reset
windows.

## 6. Field semantic clarification

`SessionSummary.messageCount` and `CountMessagesByTenantQuery` count
**user-submitted messages only**:

- ✅ Successful `conversation.send()` and `conversation.stream()` calls.
- ❌ Seed messages passed to `conversation.create()`.
- ❌ System prompts / `resolvedPrompt`.
- ❌ Assistant responses.

The event row is appended only after the underlying LLM call succeeds —
thrown provider errors do not consume quota.

## 7. Logging

Both `appendMessageEventBestEffort` (new) and `updateLastMessageBestEffort`
(retrofitted in this release) log swallowed errors at `warn` level via
`getLogger(ConversationEngine)` from `@sanamyvn/foundation/logging/global`.
Operators should monitor for `appendMessageEventBestEffort failed` and
`updateLastMessageBestEffort failed` warnings as an under-count indicator.
