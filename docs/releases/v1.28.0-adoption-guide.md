# v1.28.0 Adoption Guide ‚Äî Paginated Total Counts

> Audience: digi-talk engineering team
>
> Companion to the [release notes](./v1.28.0.md). This guide gives you
> copy-paste-ready TypeScript for building the canonical
> `{ data, meta: { total, page, perPage, totalPages } }` envelope and maps
> each change to the call sites you need to update.

## TL;DR

Four things to do when you bump to v1.28.0:

1. **Rename** `.messages` ‚Üí `.items` in every `GetSessionMessagesQuery` destructure or property access (grep recipe in ¬ß3).
2. **Read `total`** from both list responses and use it to compute `totalPages = Math.ceil(total / perPage)`.
3. **Remove pagination workarounds** if you were starting from `page: 2` to compensate for the off-by-one bug (¬ß4).
4. **Remove `TODO(ai-ts-1.27.0-count)` markers** ‚Äî the inline `total` is what those markers were waiting for (¬ß6).

No DB migration is required.

## 1. Overview

v1.28.0 adds `total: number` to both `ListSessionsQuery` and
`GetSessionMessagesQuery` responses. Both responses now share the shape
`{ items, page, perPage, total }`. With that field in hand your HTTP routes
can build the canonical pagination envelope in one SDK call without a separate
count query and without re-implementing filter logic client-side.

Two acknowledged behavior changes accompany the additive `total` field:

- `GetSessionMessagesQuery` response array is renamed `messages ‚Üí items`.
- `GetSessionMessagesQuery` `page: 1` now correctly returns the first slice of
  the thread (a v1.27.0 page-indexing bug is fixed).

Both changes require action on your side; they are called out below with exact
grep recipes and before/after snippets.

## 2. Worked example ‚Äî building `{ data, meta }`

### Sessions list

```ts
import { ListSessionsQuery } from '@sanamyvn/ai-ts/business/session/client/queries';

// In your HTTP route handler:
async function listSessions(page: number, perPage: number) {
  const { items, page: responsePage, perPage: responsePerPage, total } =
    await this.mediator.send(
      new ListSessionsQuery({ page, perPage }),
    );

  const totalPages = total === 0 ? 0 : Math.ceil(total / responsePerPage);

  return {
    data: items,
    meta: {
      total,
      page: responsePage,
      perPage: responsePerPage,
      totalPages,
    },
  };
}
```

Notes:
- `total === 0` produces `totalPages: 0`, not `NaN` or `Infinity` ‚Äî guard it explicitly.
- `items` and `total` are guaranteed to reflect the same `WHERE` clause and the
  same Postgres snapshot. You do not need to issue a separate count query.
- `SessionSummary.messageCount` is still available on each item in `items`; it
  is the user-submitted message count from the billing ledger and is unrelated
  to `total`.

### Messages list

```ts
import { GetSessionMessagesQuery } from '@sanamyvn/ai-ts/business/session/client/queries';

// In your HTTP route handler:
async function listMessages(sessionId: string, page: number, perPage: number) {
  // Note: field is now `items`, not `messages`
  const { items, page: responsePage, perPage: responsePerPage, total } =
    await this.mediator.send(
      new GetSessionMessagesQuery({ sessionId, page, perPage }),
    );

  const totalPages = total === 0 ? 0 : Math.ceil(total / responsePerPage);

  return {
    data: items,
    meta: {
      total,
      page: responsePage,
      perPage: responsePerPage,
      totalPages,
    },
  };
}
```

Notes:
- `total` here is the stored Mastra transcript count ‚Äî every message role
  (user, assistant, system, tool). It is **not** the same as
  `SessionSummary.messageCount`, which counts only successful user-submitted
  messages from the billing ledger.
- `page: 1` now returns the true first slice of the thread (bug fixed ‚Äî see ¬ß4).

## 3. Rename migration ‚Äî `messages ‚Üí items`

Every destructure or property access of `.messages` on a `GetSessionMessagesQuery`
response must become `.items`.

### Grep recipe

Run these two commands from the root of your repository. Together they catch
both destructuring patterns and property-access patterns:

```bash
# Pattern 1: destructuring ‚Äî const { messages } = ..., const { messages, page } = ...
grep -rn '{\s*[^}]*\bmessages\b[^}]*}\s*=' --include='*.ts' src/

# Pattern 2: property access ‚Äî result.messages, response.messages
grep -rn '\.\bmessages\b' --include='*.ts' src/
```

Both patterns must return zero matches in your codebase after the rename.
Run them again after your edits to confirm.

### Before / after ‚Äî destructuring

```ts
// Before (v1.27.0)
const { messages, page, perPage } = await mediator.send(
  new GetSessionMessagesQuery({ sessionId, page: 1, perPage: 50 }),
);
for (const msg of messages) { /* ... */ }

// After (v1.28.0)
const { items, page, perPage, total } = await mediator.send(
  new GetSessionMessagesQuery({ sessionId, page: 1, perPage: 50 }),
);
for (const msg of items) { /* ... */ }
```

### Before / after ‚Äî property access

```ts
// Before (v1.27.0)
const response = await mediator.send(new GetSessionMessagesQuery(...));
render(response.messages);

// After (v1.28.0)
const response = await mediator.send(new GetSessionMessagesQuery(...));
render(response.items);
```

### Before / after ‚Äî type annotation

If you annotated the response type explicitly:

```ts
// Before (v1.27.0)
import type { MessageListClient } from '@sanamyvn/ai-ts/business/session/client/schemas';
// MessageListClient had: { messages: Message[]; page: number; perPage: number }

// After (v1.28.0)
// MessageListClient now has: { items: Message[]; page: number; perPage: number; total: number }
// No import change needed ‚Äî the inferred type updated automatically with the schema.
```

## 4. Page-indexing fix

`GetSessionMessagesQuery` `page: 1` now returns the first slice of the thread,
ordered `createdAt ASC`.

In v1.27.0, a 1-based public / 0-based Mastra page-index mismatch in
`MastraMemoryAdapter` caused `page: 1` to silently read the second slice.
Mastra page 0 ‚Äî the conversation's earliest messages ‚Äî was unreachable. The fix
is a single `page - 1` subtraction in the adapter before forwarding to
`recall()`.

**What to expect after upgrading:**

- If you paginated from `page: 1`, you will now receive earlier messages that
  were previously invisible. This is the correct behavior.
- If you worked around the bug by starting pagination from `page: 2` to get
  what you thought was the "first" slice: **remove that workaround**. Starting
  from `page: 2` under v1.28.0 skips the true first slice.

**How to verify the fix in your code:** start a test thread, send 3 messages,
fetch `page: 1, perPage: 2`. The response `items[0]` should be the first
message you sent, not the second.

## 5. Snapshot caveat

`total` and row positions may drift between pages if writes occur between calls.
This is the same isolation semantics as v1.27.0 ‚Äî `total` just makes the
existing drift more observable.

Practical consequences:

- A multi-page UI scroll may show a `total` that nudges between page fetches if
  messages are being appended concurrently.
- A UI displaying `totalPages` derived from an early-page `total` may show a
  stale page count by the time the user reaches the last page.

The package does not offer a transactional read mode. If you need a stable
snapshot for a specific use case (e.g. export), issue a single oversized page
(`perPage: 500`) or implement client-side reconciliation by message id. See
[design doc ¬ß3 "Snapshot caveat"](../design/paginated-total-counts.md#snapshot-caveat)
for the full analysis.

## 6. Verification checklist

Work through these in order before merging your digi-talk adoption branch:

- [ ] Bump `@sanamyvn/ai-ts` to `^1.28.0` in your `package.json` and run
  `pnpm install` (or `npm install` / `yarn`).
- [ ] Run the grep recipe from ¬ß3 against your `src/` directory. Zero matches
  for both patterns.
- [ ] Update all matching destructures and property accesses from `.messages`
  to `.items`.
- [ ] Remove any `page: 2` workarounds that were compensating for the
  page-indexing bug (¬ß4).
- [ ] Wire `total` into your canonical envelope helpers:
  `totalPages = total === 0 ? 0 : Math.ceil(total / perPage)`.
- [ ] Run your full test suite. Confirm it passes.
- [ ] Remove `TODO(ai-ts-1.27.0-count)` markers ‚Äî these were placeholders for
  inline `total`, which has now landed. The canonical locations in digi-talk:
  - `client/ai-session.queries.ts` ‚Äî limitations comment next to the
    `ListSessionsQuery` wrapper, if still present.
  - Any route handler that currently hard-codes `totalPages: null` or derives
    it with a client-side `sessions.length` approximation.
  - Any test that asserts `meta.total` is `undefined` or skips the assertion
    entirely.
- [ ] Confirm `SessionSummary.messageCount` and `CountMessagesQuery`
  still behave identically ‚Äî no change is required for the billing path.
- [ ] Deploy SDK and server together in the same window. Mixed-version
  deployments (v1.28.0 SDK ‚Üî v1.27.0 server or vice versa) will produce Zod
  parse errors on the remote mediator. There is no v1.27.0 maintenance branch.
