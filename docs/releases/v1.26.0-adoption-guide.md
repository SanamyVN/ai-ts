## 2. Ask 1 — `messageCount` on `SessionSummary`

Added to the client schema and the response DTO. Backed by the new
`ai_session_messages` event ledger; the session list path issues one
`countBySession` query per page and projects the count onto each summary —
no per-summary round trip.

```ts
import { ListSessionsQuery } from '@sanamyvn/ai-ts/business/session/client/queries';

const { items } = await this.mediator.send(
  new ListSessionsQuery({ tenantId, page: 1, perPage: 50 }),
);

for (const summary of items) {
  console.log(summary.id, summary.messageCount); // number, lifetime, ≥ 0
}
```

**Important:** `messageCount` is **lifetime**, not period-bounded. Sessions that
have messages outside the requested period still report the full count. For
period-bounded billing, use `CountMessagesByTenantQuery` (§5).

Counted events: only successful `conversation.send()` / `conversation.stream()`
calls. Excluded: seed messages, system prompts, assistant responses, failed LLM
calls.

## 3. Ask 2 — Time-window filter on `ListSessionsQuery`

You drafted `createdAtGte` / `createdAtLt`; we shipped `startedAtGte` /
`startedAtLt`. The schema column is `ai_sessions.started_at` (existed before
this release) — we kept the name aligned to avoid two terms for the same field.

```ts
new ListSessionsQuery({
  tenantId,
  startedAtGte: periodStart, // include sessions started on or after periodStart
  startedAtLt:  periodEnd,   // exclude sessions started on or after periodEnd
  page: 1,
  perPage: 100,
});
```

Semantics:
- Half-open interval `[Gte, Lt)`.
- When both are present, server enforces `startedAtLt > startedAtGte` and
  rejects the call with a Zod refinement message
  `'startedAtLt must be strictly greater than startedAtGte'`.
- HTTP DTOs require `z.iso.datetime({ offset: true })` — datetimes must carry
  an explicit offset (e.g. `2026-04-01T00:00:00Z`). Offset-less strings are
  rejected at the boundary to prevent server-timezone drift.

## 4. Ask 3 — `tenantId` on `IConversationEngine.create()`

`ConversationConfig.tenantId?: string` was already part of the interface in
prior releases — only the JSDoc and example were missing. v1.26.0 closes that
documentation gap. No code change required on your end beyond passing the
field.

```ts
await this.conversationEngine.create({
  tenantId: this.asyncCtx.require(TENANT_ID),
  promptSlug: 'teaching-assistant',
  promptParams,
  userId,
  purpose: `ta-chat:${taId}`,
  metricsContext: { /* ... */ },
});
```

Behavior:
- When `tenantId` is provided, it is persisted to `ai_sessions.tenant_id`.
- When omitted, the session is **tenantless** (`tenant_id` stored as `null`).
  Tenantless sessions are intentionally **not billable** — message events for
  them are skipped (the `appendMessageEventBestEffort` hook returns silently
  when `session.tenantId === null`). If you forget to thread `tenantId` for a
  call site, you will under-count rather than over-count.
- Backward compatible: existing call sites that omit `tenantId` continue to
  work — they just don't bill.

## 5. Recommended billing path — `CountMessagesByTenantQuery`

Your feature request planned this:

```ts
// Plan from feature request §"Proposed implementation"
const sessions = await this.mediator.send(
  new ListSessionsQuery({
    tenantId,
    purposePrefix,
    startedAtGte: periodStart,
  }),
);
const count = sessions.reduce((sum, s) => sum + s.messageCount, 0);
```

**Use this instead:**

```ts
import { CountMessagesByTenantQuery } from '@sanamyvn/ai-ts/business/session/client/queries';

const { count } = await this.mediator.send(
  new CountMessagesByTenantQuery({
    tenantId,
    purposePrefix: 'ta-chat:',
    sentAtGte: periodStart,
    sentAtLt:  periodEnd,
  }),
);
```

Why `CountMessagesByTenantQuery` is the right billing aggregate:

| Concern | `sum(messageCount)` over `ListSessionsQuery` | `CountMessagesByTenantQuery` |
|---------|---------------------------------------------|------------------------------|
| Filters by message send time | ❌ Filters by session start time | ✅ Filters on `sent_at` |
| Period-spanning sessions | ❌ A session started in March with messages in April lands in March's bucket | ✅ Each event lands in its own period |
| Round trips | One per page until pagination exhausts | One query |
| Index path | Composite per filter | `(tenant_id, sent_at)` composite scan |
| Matches lifetime semantics | Yes (sums lifetime counts) | No — period-bounded |

`startedAtGte` / `startedAtLt` on `ListSessionsQuery` use **session start time**.
`sentAtGte` / `sentAtLt` on `CountMessagesByTenantQuery` use **message send
time**. They are deliberately different field names — see §6.7 of the design
doc for the anti-conflation guarantee.

Same Zod refinements: `purpose` and `purposePrefix` are mutually exclusive;
`sentAtLt > sentAtGte` when both are present.

## 6. Breaking changes that affect your call sites

Three side effects of this release that touch your existing code:

1. **`ListSessionsQuery` response shape**: `SessionSummary[]` →
   `{ items: SessionSummary[]; page: number; perPage: number }`. Iterate over
   `result.items` instead of `result`.
2. **`ListSessionsQuery` requires pagination**: `page` ≥ 1 and `perPage` 1–500
   are mandatory. No silent default.
3. **`GetSessionMessagesQuery`**: `page` ≥ 1, `perPage` 1–500.

If you don't have natural pagination from a UI page, supply `page: 1, perPage: 500`
and iterate. For aggregate use cases, prefer `CountMessagesByTenantQuery`
(§5) — no pagination needed.

## 8. Backfill semantics — first-period accuracy gap

`ai_session_messages` ships empty. Pre-existing sessions report
`messageCount: 0` until their next user message; aggregate counts over
historical periods are zero.

**The first full period after deploy is the first accurate one.**

If you bill on partial periods, the partial period that wraps the deploy
window will under-count. Surface this to the entitlement engine:

- Display-only usage indicators: include a "data available since YYYY-MM-DD"
  stamp.
- Hard quota gates: do not enforce against the partial period covering the
  deploy.

There is no backfill tool. If retroactive accuracy matters, write a one-shot
migration that walks `mastra_messages` (or your conversation history source)
and inserts synthetic ledger rows.

## 9. Logging hooks for ops monitoring

Both best-effort helpers in `ConversationEngine` log swallowed errors at
`warn` via `getLogger('ConversationEngine')` (`@sanamyvn/foundation/logging/global`):

- `appendMessageEventBestEffort failed` — ledger write rejected after a
  successful LLM call. Treat as **under-count**: tenant-favorable, not
  over-count, but worth a dashboard.
- `updateLastMessageBestEffort failed` — sessions list will not reflect the
  most recent message preview for that session.

Both carry structured context: `{ sessionId, err: Error }`. Add to your log
search.

## 10. Trust boundary on the new HTTP routes

v1.26.0 adds two routes in the package router:

- `POST /ai/sessions/:id/message-events` (writes the billing ledger)
- `GET  /ai/sessions/message-events/count` (reads the billing aggregate)

Today digi-talk does **not** mount `SessionAppModule` — these routes are
unreachable to tenants. Local mediator dispatch handles the in-process hook;
no HTTP traffic.

If a future deployment ever mounts `SessionAppModule` (e.g. microservice
topology, internal admin surface), wire `SESSION_MIDDLEWARE_CONFIG.appendMessageEvent`
and `.countMessagesByTenant` to your `StsAuthMiddleware` (or equivalent
service-to-service auth). The `sentAt` field is caller-supplied by design
(captures user submission time, not stream completion time) — without S2S
auth, a tenant who can reach the route can manipulate billing periods.

## 11. Recommended indexes (you decide when to ship)

Not shipped in this release; promote from "recommended" to "ship" once your
ledger volume crosses ~10K rows per tenant per month:

```sql
CREATE INDEX ai_session_messages_session_idx
  ON ai_session_messages (session_id);

CREATE INDEX ai_session_messages_tenant_sent_idx
  ON ai_session_messages (tenant_id, sent_at);

CREATE INDEX ai_session_messages_tenant_purpose_sent_idx
  ON ai_session_messages (tenant_id, purpose, sent_at);
```

`session_idx` is the most important — `SessionRepository.list` makes a
per-page `countBySession` query that scans this index. Without it, list pages
will degrade linearly with ledger size.

## 12. Verification before flipping the proxy

Before promoting `CountMessagesByTenantQuery` to a billing input or hard quota
gate:

1. Deploy v1.26.0 to staging and run for one full billing period.
2. Compare `CountMessagesByTenantQuery` output against the proxy
   (`sessions.length`) for the same tenant — ratio should diverge as expected
   (sessions vs. messages).
3. Validate that the period-spanning case bills correctly: create a session
   on day `D-1` of period boundary; send messages on `D-1`, `D`, `D+1`;
   confirm the events split across periods.
